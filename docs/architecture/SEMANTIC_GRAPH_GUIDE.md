# Semantic Graph — руководство

> Прототип графового расширения поиска для ChatAVG v2.3.
> Статус: **opt-in**, только для тарифа «Мудрец» (sage).

---

## 1. Что такое Semantic Graph (концепция)

**Semantic Graph** — это граф понятий и связей, извлечённых из корпуса знаний системы. Узлы графа представляют сущности, концепты, домены и секции документов; рёбра — отношения между ними (упоминание, вложенность, ассоциация).

### Назначение

Обычный поиск (лексический + векторный) находит **похожие** фрагменты — те, в которых слова или эмбеддинги совпадают с запросом. Semantic Graph находит **связанные** фрагменты — даже если они не содержат слов из запроса, но связаны с найденными через общие сущности.

| Подход | Принцип | Результат |
|--------|---------|-----------|
| Обычный поиск | «Найди похожее» | Фрагменты, похожие на запрос |
| Semantic Graph | «Найди связанное» | Фрагменты, связанные с найденными через общие сущности |

### Аналогия

Представьте, что вы ищете книгу по слову «бытийность». Обычный поиск вернёт фрагменты, где это слово встречается. Semantic Graph дополнительно найдёт фрагменты, где обсуждается «метареальность» или «эзоагностика реальности», потому что эти понятия связаны в графе через общие рёбра.

### Текущий статус

Прототип доступен **опционально** и только для тарифа **Мудрец** (sage). Включается через переменную окружения или параметр категории.

## 2. Решение Go/No-Go (S8-5)

По итогам спайка, завершённого **2026-06-09**, принято следующее решение:

| Решение | Обоснование |
|---------|-------------|
| **NO-GO** — обязательное включение в RAG v1 | Система полностью работоспособна без Semantic Graph; внедрение в v1 добавит ненужную сложность |
| **GO** — опциональный прототип (sage only) | Минимальный риск, возможность оценки пользы на реальных данных |

### Критерии оценки

| Критерий | Результат спайка | Вердикт |
|----------|-------------------|---------|
| Качество извлечения сущностей | На основе правил (глоссарий + regex). Достаточно для спайка, **не** достаточно для auto-ingest v1 | Отложить улучшение до v2 |
| Задержка графового расширения (1-hop) | In-process PG lookup, без LLM — быстро | Приемлемо |
| Прирост полноты (recall lift) vs metadata-only | Не измерен офлайн | Отложено до v2 |
| Операционная сложность | +2 PG-таблицы, spike-скрипт, opt-in флаг | Приемлемо для v2 |

### Результаты спайка

- Обработано: **5 книг × 4 чанка = 20 чанков**
- Всего сущностей: **257** (уникальных: **151**)
- Среднее количество сущностей на чанк: **12.85**
- Покрытие доменными тегами (`domainTagCoverage`): **1.0**

## 3. Как это работает (для пользователей)

Когда пользователь задаёт вопрос в категории **«Мудрец»** и Semantic Graph включён, поиск проходит через 4 этапа:

### Этап 1: Векторный поиск

Система выполняет обычный векторный + лексический поиск, как и в других тарифах. Результат — набор релевантных чанков с базовыми оценками.

### Этап 2: Идентификация сущностей

Для каждого найденного чанка система определяет, какие сущности (понятия, термины) в нём упоминаются. Извлечение выполняется на основе правил: глоссарий терминов (`«бытийность»`, `«метареальность»`, `«эзоагностика реальности»` и др.), regex для Title Case и аббревиатур.

### Этап 3: Графовое расширение

Система обходит граф на 1 шаг (1-hop) от найденных сущностей и определяет соседние чанки, связанные через общие узлы. Расширенные чанки получают оценку с понижающим коэффициентом:

```
score = baseScore × EXPANSION_SCORE_DECAY × edgeWeight
```

где `EXPANSION_SCORE_DECAY = 0.85`. Это гарантирует, что расширенные результаты не вытесняют прямые попадания.

### Этап 4: Обогащение контекста

Расширенные чанки добавляются в контекст ответа. В цитатах появляется маркер `graph_expanded: true`, показывающий, что данный фрагмент найден через графовое расширение, а не прямым поиском.

### Резюме потока

```
Запрос пользователя
       │
       ▼
  Векторный поиск ──→ начальные чанки
       │
       ▼
  Извлечение сущностей из чанков
       │
       ▼
  1-hop обход графа ──→ соседние чанки (score × 0.85)
       │
       ▼
  Обогащённый контекст ──→ ответ LLM
```

## 4. Варианты использования (use cases)

| Сценарий | Описание | Пример |
|----------|----------|--------|
| **Многошаговые вопросы** | Концепции из разных книг/глав связаны через граф | Вопрос о «бытийности» находит не только прямые упоминания, но и связанную дискуссию о «метареальности» из другой книги |
| **Глубокий анализ** | Все упоминания концепта во всём корпусе | Запрос о «структурах сознания» находит все чанки, где эта сущность присутствует, даже если слова из запроса там нет |
| **Перекрёстные ссылки** | Определение в одной книге, примеры — в другой | Определение термина найдено напрямую, а примеры его использования — через графовую связь |
| **Составное исследование** | Последовательные вопросы с углублением | Первый запрос даёт базовый контекст, последующие — расширяют через связи, не теряя нить рассуждения |

## 5. Инструкция для администратора

### Включение / выключение

Semantic Graph можно включить двумя способами:

**Способ 1: Переменная окружения (глобально для sage)**

```bash
SEMANTIC_GRAPH_ENABLED=true
```

Устанавливается в `.env` или переменных среды сервера. Действует для всех запросов тарифа «Мудрец».

**Способ 2: Параметр категории (точечно)**

В настройках категории (поле `extra_params`) добавить:

```json
{
  "semantic_graph_enabled": true
}
```

Приоритет: `extra_params.semantic_graph_enabled` > `process.env.SEMANTIC_GRAPH_ENABLED`.

> **Важно:** Функция активируется **только** для тарифа `sage`. Для других тарифов флаг игнорируется. Логика разрешения: `resolveSemanticGraphEnabled(tier, extraParams, env)` — возвращает `true` только при `tier === 'sage'`.

### Требования для работы

| Компонент | Описание |
|-----------|----------|
| PG-таблица `kb_semantic_nodes` | Узлы графа (id, namespace, node_type, label, canonical_key, doc_id, chunk_id, metadata) |
| PG-таблица `kb_semantic_edges` | Рёбра графа (id, namespace, source_node_id, target_node_id, edge_type, weight, metadata) |
| Заполненные `entity_ids` | Векторные чанки должны содержать поле `entity_ids` со связанными сущностями |
| Заполненный граф | Таблицы nodes/edges должны содержать данные (заполняются через отдельный скрипт индексации) |

### Мониторинг

- **Trace-события:** в логах запроса отражается факт графового расширения и количество добавленных чанков.
- **Деградированный режим:** если граф недоступен (таблицы пусты / ошибка PG), система автоматически возвращается к обычному поиску без расширения. Пользователь не видит ошибку — ответ формируется на основе прямых попаданий.

## 6. Ограничения текущей версии

| Ограничение | Описание |
|-------------|----------|
| **Извлечение сущностей — только на основе правил** | Не используется LLM; извлечение опирается на глоссарий, regex (Title Case) и аббревиатуры. Это ограничивает покрытие и точность. |
| **Только 1-hop расширение** | Обход графа ограничен одним шагом. Многошаговые связи не раскрываются. |
| **Только тип рёбер `mentions`** | Таблица `kb_semantic_edges` содержит только рёбра типа `mentions`. Типы `part_of`, `related_to`, `same_domain` определены в схеме, но не заполняются. |
| **Нет административного UI** | Управление графом (просмотр, очистка, переиндексация) — только через скрипты и SQL. |
| **`entity_ids` не заполняются при ingest** | Поле `entity_ids` в чанках не заполняется автоматически при загрузке документов. Требуется отдельная переиндексация (скрипт спайка). |

## 7. Дорожная карта и план реализации v2

### Обзор этапов

```
Этап 1: Foundation & Eval      (Спринт SG-1, 2 недели)
Этап 2: LLM Entity Linking    (Спринт SG-2, 2 недели)
Этап 3: Production Hardening  (Спринт SG-3, 2 недели)
Этап 4: Interface & Admin     (Спринт SG-4, 1–2 недели)

Итого: ~8 недель (1 dev)
```

---

### Этап 1 — Foundation & Eval (Спринт SG-1, 2 недели)

**Цель:** Заполнить `entity_ids` в корпусе, измерить recall lift, принять go/no-go для полного включения.

| ID | Задача | DoD |
|----|--------|-----|
| SG1-1 | Расширить `IngestionPipeline`: вызов `entityExtractionService.extract()` → запись `entity_ids` в `kb_chunks` + upsert nodes/edges | Unit + integration test |
| SG1-2 | Re-index канонического корпуса с entity extraction (полный корпус) | `kb_chunks.entity_ids` populated; `kb_semantic_nodes` count > 500 |
| SG1-3 | Offline eval: recall@k с graph expansion vs metadata-only (≥30 запросов из eval set S0-7) | Report: recall@3, recall@7, MRR; delta vs baseline |
| SG1-4 | Performance benchmark: 1-hop expansion latency на populated graph | p95 < 100ms |
| SG1-5 | **Go/No-Go gate:** recall lift < 5% → defer; ≥5% → proceed | Decision doc |

**Критерий выхода:** entity_ids заполнены, eval проведён, gate пройден/отклонён.

---

### Этап 2 — LLM-Assisted Entity Linking (Спринт SG-2, 2 недели)

**Цель:** Повысить качество entity extraction с помощью LLM.

| ID | Задача | DoD |
|----|--------|-----|
| SG2-1 | `LlmEntityExtractor`: prompt-based NER (GPT-4.1-mini / local) | Interface + implementation + unit test |
| SG2-2 | Hybrid pipeline: rule-based (fast) + LLM (batch, async) при ingest | Config: `ENTITY_EXTRACTION_MODE=rule\|llm\|hybrid` |
| SG2-3 | Edge type enrichment: `part_of`, `related_to`, `same_domain` | Populated in re-index; integration test |
| SG2-4 | Eval: rule-based vs LLM extraction quality (precision/recall) | Report |
| SG2-5 | Cost model: LLM extraction per 1K chunks | Budget doc |

**Критерий выхода:** LLM extractor работает, разнообразные edge types заполнены, стоимость оценена.

---

### Этап 3 — Production Hardening (Спринт SG-3, 2 недели)

**Цель:** Graph expansion готов для production sage tier.

| ID | Задача | DoD |
|----|--------|-----|
| SG3-1 | Graph GC: orphan cleanup при удалении документа | Integration test |
| SG3-2 | Multi-hop expansion (2-hop для sage): рекурсивный expand | Unit test + p95 < 200ms |
| SG3-3 | Graph-aware scoring: вес edge_type в decay (mentions < part_of < related_to) | Configurable weights |
| SG3-4 | Load test: 10 concurrent sage queries, 5K+ nodes | p95 < 300ms (NFR-1) |
| SG3-5 | Circuit breaker: graph timeout → fallback to metadata-only | Auto-fallback test |
| SG3-6 | Observability: `rag.graph_expand_ms`, `rag.graph_nodes_traversed`, `rag.graph_fallback` | traceBus events |

**Критерий выхода:** graph expansion стабилен под нагрузкой, degraded mode работает, metrics в trace.

---

| ID | Задача | DoD |

### Критерии Go/No-Go для полного включения (после Этапа 1)

| Критерий | Порог | Метод проверки |
|----------|-------|----------------|
| Recall@7 lift (graph vs no-graph) | ≥ 5% абсолютный | Offline eval на S0-7 |
| 1-hop p95 latency | ≤ 150ms | Load test |
| Entity extraction precision | ≥ 70% | Manual review 50 chunks |
| Operational overhead | ≤ 2 extra PG queries/request | Trace analysis |
| Tenant isolation | 0 leaks | Automated E2E tests |
